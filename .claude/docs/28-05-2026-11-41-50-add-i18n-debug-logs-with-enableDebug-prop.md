## Approach Summary

- Instrumentation-only plan. We do NOT yet fix the suspected root cause (i18next's `addResourceBundle` runtime mutation does not notify mounted `useTranslation` subscribers, so already-rendered screens keep the pre-merge `t` snapshot). The instrumentation is designed to confirm this hypothesis in a real browser session before any structural fix is planned.
- A new optional component prop `enableDebug?: boolean` (default `true`) replaces the previous `import.meta.env.DEV` gate. Because logs originate in three places (the standalone `applyI18nConfig` module function, the `<FaceRegister>` component, and the three organism screens), we need a small contract that lets the standalone module see the same flag the component sees.
- Chosen mechanism for module-level gating: a module-scoped `let isI18nDebugEnabled = true` in `src/shared/i18n/index.ts` plus a tiny exported setter `setI18nDebugEnabled(enabled: boolean): void`. `FaceRegister` calls the setter synchronously, in the same place it already calls `applyI18nConfig` (the `i18nConfigRef` first-render block and the `useLayoutEffect`). This keeps `applyI18nConfig`'s signature untouched (no signature churn at the public-ish module boundary), keeps the blast radius to one file, and the setter is private (NOT re-exported from `src/index.ts`) so it does not enlarge the public API surface. Trade-off documented below in Risks.
- Organism screens accept an optional `enableDebug?: boolean` prop drilled from `FaceRegister`. Their internal helper consults the prop directly, so no module-level state is needed at the React layer. Default at every layer is `true`, mirroring the public default.
- All log lines use the literal prefix `[i18n-debug]` so a single `grep -rn "\[i18n-debug\]" src/` removes them in one pass when the bug is fixed.

## Functional Requirements

- `<FaceRegister>` accepts a new optional prop `enableDebug?: boolean`. Default value: `true`.
- When `enableDebug === true`, every layer below emits a `[i18n-debug]` line at the moments listed below. When `enableDebug === false`, zero `[i18n-debug]` output is produced from any layer.
- `src/shared/i18n/index.ts`:
  - Holds module-scoped `isI18nDebugEnabled` (default `true`) and exports `setI18nDebugEnabled(enabled: boolean): void`.
  - Inside `applyI18nConfig`, when the flag is on, logs:
    - Inputs (`locale`, presence and keys of `translations`).
    - Pre-state: `libraryI18n.language`, `isInitialized`, and the current `faceRegister` resource bundle for the active language (a snapshot via `getResourceBundle`).
    - Each branch taken (built-in bundle re-added; `changeLanguage` called; consumer `translations` bundle added) with arguments.
    - Post-state: `libraryI18n.language` and the post-merge `faceRegister` resource bundle for the active language.
- `src/templates/FaceRegister/index.tsx`:
  - Destructures `enableDebug = true` from props.
  - Calls `setI18nDebugEnabled(enableDebug)` synchronously **before** the first `applyI18nConfig(locale, translations)` call in the `i18nConfigRef` first-render block, AND inside the `useLayoutEffect([locale, translations])` before the second `applyI18nConfig` call, so the setter is always in sync with the prop when logs fire.
  - When `enableDebug === true`, logs at first-render apply, at every `useLayoutEffect` apply (with the prop values that triggered the re-apply), and the resolved `t("faceRegister.introStep1")` value returned by a one-shot `libraryI18n.t(...)` call inside the `useLayoutEffect` (read-only — does not change behaviour).
  - Drills `enableDebug` down to `<IntroScreen>`, `<CaptureScreen>`, and `<ResultScreen>`.
- `src/organisms/IntroScreen/index.tsx`, `src/organisms/CaptureScreen/index.tsx`, `src/organisms/ResultScreen/index.tsx`:
  - Each accepts a new optional `enableDebug?: boolean` (default `true`) on its props interface.
  - On render, when `enableDebug === true`, logs the screen name, `libraryI18n.language`, and the values of the user-visible keys it renders (at minimum: `IntroScreen` logs `introStep1`/`introStep2`/`introStep3`/`introTitle`/`introSub`; `CaptureScreen` logs `hudTitle` and the current `step.instructionKey`; `ResultScreen` logs `resultTitle`/`resultSub`).
  - Logging is performed via a small inline helper (e.g. `const log = enableDebug ? console.log : noop`) — no separate utility module.
- Every log line begins with the literal prefix `[i18n-debug]` followed by a short scope tag (e.g. `[i18n-debug][applyI18nConfig:pre]`, `[i18n-debug][FaceRegister:mount]`, `[i18n-debug][IntroScreen:render]`).
- Public type `FaceRegisterProps` (`src/shared/types/faceRegister.ts`) is extended with `enableDebug?: boolean` and a JSDoc note that the default is `true` and that consumers should set `false` to silence debug logs in production.
- Logs are emitted into the browser DevTools console when reproducing per the repro steps in the delegation.
- After collecting logs, a follow-up plan (separate delegation, not in scope here) will either confirm the subscriber-refresh root cause and propose a structural fix, or pivot if the logs disprove it.

## Non-Functional Requirements

- Zero behavioural change when `enableDebug === false`: prop flow, render output, and React effect timing must be identical to current behaviour. The only added work in that mode is property reads, prop drilling, and a single `setI18nDebugEnabled(false)` call per `applyI18nConfig` invocation.
- Zero behavioural change when `enableDebug === true` either, beyond logging side-effects — no extra renders, no extra effects, no extra subscriptions.
- All instrumentation is removable in a single grep pass: `grep -rn "\[i18n-debug\]" src/` returns every site to delete. The `enableDebug` prop, the module-scoped flag, and `setI18nDebugEnabled` must also be removable in the same teardown step and are explicitly documented as such inline (single JSDoc line above each).
- TypeScript strict mode and `verbatimModuleSyntax` must remain clean — `import type` for type-only imports, no `any`.
- Follow existing project conventions: 2-space indent, double quotes, semicolons, trailing commas, default-exported components, `<Component>Props` interface above each component, hooks return objects, relative imports.
- No new runtime dependency. Use only `console.log` and the already-present `libraryI18n` API surface (`getResourceBundle`, `t`, `language`, `isInitialized`).
- `setI18nDebugEnabled` is internal — it is NOT added to `src/index.ts` and NOT added to `tsconfig.build.json`'s declaration emit. Only `enableDebug` (via `FaceRegisterProps`) becomes part of the published API surface.
- Logged values must not include any user PII; for this codebase the only logged values are i18n keys / translation strings / language codes, which is safe.

## Risks & Assumptions

- **Advisory root cause (NOT bundled into tasks per delegation constraint):** the most likely reason the override is dropped is that `libraryI18n.addResourceBundle(...)` mutates the resource store but does NOT trigger an `i18next` re-render event for already-mounted `useTranslation` subscribers. Since `<IntroScreen>` mounts on the initial render — the same render in which `applyI18nConfig` is called from the first-render `if (!i18nConfigRef.current)` block — the `t()` snapshot it captures may pre-date the merge by a microtask, depending on `useTranslation`'s subscription order. The instrumentation is designed to confirm or refute this. A structural fix (e.g. call `libraryI18n.changeLanguage(currentLanguage)` after a bundle add to force a `languageChanged` event, or migrate from `addResourceBundle` to a startup-time merge into `resources`) belongs in the next delegation, not this one.
- **`enableDebug` default of `true` ships noise to consumers.** This is the user's explicit preference, so the plan does not change it. The implication is that any application installing `@ngmthaq20/react-face-id-capture` will see `[i18n-debug]` lines in the browser console on every render of `<FaceRegister>` and on every i18n apply, unless they pass `enableDebug={false}`. README does not currently document the prop, and the user did not ask for README changes, so documentation is left out of scope — but a follow-up task to document the new prop in the README before the next npm publish is recommended.
- **New public API surface.** `enableDebug` enters `FaceRegisterProps` which is re-exported from `src/index.ts` and will appear in `dist/index.d.ts` after the next `yarn build:lib`. Once shipped, removing or renaming the prop is a breaking change. The plan accepts this because the user requested it as a component prop; alternatives (env-var, global toggle, debug build) were not requested.
- **Module-scoped `let isI18nDebugEnabled` + `setI18nDebugEnabled` is shared mutable state.** If a host app mounts two `<FaceRegister>` instances with conflicting `enableDebug` values, whichever instance most recently called `setI18nDebugEnabled` wins for the duration. This is acceptable because `<FaceRegister>` is a full-screen capture template not designed for multi-instance use, but it is a real trade-off versus threading an explicit parameter through `applyI18nConfig`. Documented inline in `src/shared/i18n/index.ts` with a one-line JSDoc.
- **Alternative considered and rejected:** adding an `enableDebug?: boolean` parameter to `applyI18nConfig` would avoid module state but would change the signature of an exported function and would require every caller to remember the flag — higher blast radius and easier to forget at teardown.
- **`console.log` chattiness on every render.** When `enableDebug === true`, the three organisms log on every render, which can be noisy during a capture session (where re-renders fire on each detection tick). This is acceptable for a debug session, and is precisely the signal we want to spot stale `t()` output, but the user should know to filter the DevTools console by `[i18n-debug]` to keep it readable.
- **Assumption:** `libraryI18n.getResourceBundle(lng, "translation")` returns the live merged store object at call time (per the i18next API). If a future i18next upgrade changes that semantic, the pre/post snapshot logic will need to be re-validated. No upgrade is part of this plan.
- **Assumption:** logs in the browser console are sufficient to diagnose the issue. If they are not, a follow-up plan adds an `onDebug?: (event) => void` callback prop so the host app can capture log events programmatically; not in scope here.
- **No regression test is included in this plan.** Per project `PROJECT_OVERVIEW.md`, testing workflow is `Skip-Testing` and there is no test framework configured. The debugger skill's "regression test mandatory" rule is reinterpreted here as: the regression check is the **manual reproduction** described in the Repro Verification task below. If the user wants real automated tests, that requires bootstrapping a test framework (Vitest + RTL would be the obvious choice) and is a separate delegation.
